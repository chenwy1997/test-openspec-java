# /sql-risk-review — SQL 风险审查

## 触发方式

用户输入 `/sql-risk-review` 时执行本技能。

## 功能说明

专项检查本次变更中涉及数据库操作的代码和 SQL，识别可能导致数据事故的高风险写法。

参考规范：`docs/standards/database.md`

## 执行步骤

1. 读取 `docs/standards/database.md` 获取完整数据库规范
2. 执行 `git diff HEAD -- "*.xml" "*.java" "*.sql"` 获取数据库相关变更
3. 按以下维度逐一检查

## 审查维度

### SQL 注入风险
- [ ] MyBatis XML 中是否有 `${}` 直接拼接用户输入
- [ ] 动态 `ORDER BY ${column}` 是否经过白名单校验

### 软删除遗漏
- [ ] 手写 SQL（非 MyBatis Plus 自动注入的查询）是否加了 `deleted = 0` 条件
- [ ] 更新/删除操作是否确认只操作未删除的数据

### 批量操作安全
- [ ] 所有 `UPDATE ... SET` 操作是否有 `WHERE` 条件
- [ ] 批量 `DELETE` 是否有 `WHERE` 条件
- [ ] 批量更新的影响行数是否有上限控制（`LIMIT`）
- [ ] `IN (ids)` 的 id 列表是否有数量上限保护

### 索引与性能
- [ ] 查询条件中的字段是否有索引
- [ ] 是否有 `LIKE '%xxx'` 开头模糊查询（无法走索引）
- [ ] 是否有在 WHERE 条件中对字段做函数运算（如 `DATE(created_at) = ?`）
- [ ] 新增的慢查询是否有评估

### 字段类型规范
- [ ] 金额字段是否使用了 `DECIMAL`（Java 层是否用 `BigDecimal`）
- [ ] 是否有用 `double`/`float` 存储金额（极度危险）

### 事务安全
- [ ] 是否有捕获异常但不重新抛出（导致事务不回滚）
- [ ] 是否有长事务（事务内有远程调用或复杂计算）
- [ ] `@Transactional` 是否加在 Controller 层（禁止）

### DDL 风险（如有 DDL 变更）
- [ ] 修改字段类型是否评估了锁表影响
- [ ] 删除字段是否确认代码层已同步移除引用
- [ ] 新增索引是否评估了对写入性能的影响

## 风险等级定义

| 等级 | 含义 | 示例 |
|------|------|------|
| 🔴 高危 | 可能直接导致数据丢失或安全漏洞 | SQL 注入、批量无 WHERE 更新 |
| 🟡 中危 | 可能在高并发或大数据量下出问题 | 无索引的全表扫描、未加 LIMIT 的批量操作 |
| 🟢 低危 | 代码规范问题，不影响数据安全 | 字段命名不规范、缺少注释 |

## 输出格式

```markdown
## SQL 风险审查报告

**审查范围**：[本次变更中涉及数据库操作的文件列表]

### 高危风险 🔴（必须修复）

> 如无，写"无高危风险"

**[风险类型]** `文件路径:行号`
- 风险描述：[具体说明]
- 潜在影响：[如果不修复会发生什么]
- 修复建议：[如何修复]

---

### 中危风险 🟡（强烈建议修复）

> 如无，写"无中危风险"

- `文件路径:行号` — [说明]

---

### 低危/规范问题 🟢（可在下次迭代处理）

> 如无，写"无"

- [说明]

---

### 整体评估

- **数据安全风险**：高 / 中 / 低
- **性能风险**：高 / 中 / 低
- **建议**：[是否可以合并，还是必须先处理]
```
