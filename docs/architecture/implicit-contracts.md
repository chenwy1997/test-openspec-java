## 通用约定

### 软删除约定
**约定**：本项目所有表均使用软删除，`deleted` 字段标记删除状态。`deleted = 0` 为正常，`deleted = 1` 为已删除。

**风险**：查询时忘记加 `deleted = 0` 条件，会查出已删除数据。MyBatis 全局 SQL 拦截器会自动追加此条件，但**手写 XML SQL 时必须手动加**。

**示例**：
```sql
-- ✅ 正确（手写 SQL 时）
SELECT * FROM t_user WHERE id = #{id} AND deleted = 0

-- ❌ 错误（手写 SQL 忘记软删除条件）
SELECT * FROM t_user WHERE id = #{id}
```

---

### status 字段语义
**约定**：`status = null` 与 `status = 0` 在历史数据中**语义不等价**。

- `status = null`：历史遗留数据，表示"初始化未完成"状态
- `status = 0`：明确的"禁用/关闭"状态
- `status = 1`：正常/启用状态

**风险**：使用 `status != 1` 查询"非正常"数据时，会把 `null` 值数据也包含进来，可能引发错误的业务判断。

**正确处理**：
```java
// 判断是否"正常"
if (entity.getStatus() != null && entity.getStatus() == 1) { ... }

// 查询正常数据的 SQL（必须显式判断 null）
WHERE status = 1
```

---

### 分页约定
**约定**：所有列表接口必须支持分页，禁止不加 LIMIT 的全表查询返回给前端。

**风险**：数据量大时直接压垮接口。

**默认值**：`pageNum = 1`，`pageSize = 20`，`pageSize` 最大不超过 100（超出则强制截断为 100）。

## 用户模块

### 用户手机号唯一性
**约定**：手机号在系统中唯一，但软删除的用户不参与唯一性校验（即同一手机号可以注册新账号，如果历史账号已软删除）。

**风险**：注册时的唯一性校验 SQL 必须加 `deleted = 0` 条件。

---

## 前端接口约定

### 接口向后兼容
**约定**：已上线接口的响应字段**只能新增，不能删除或重命名**。前端可能已经硬编码了某些字段名。

**风险**：删除或重命名响应字段会导致前端展示异常。

**正确做法**：废弃字段保留并标记 `@Deprecated`，新字段用新名称添加。

---

## 数据库约定

### 时间字段
**约定**：所有表必须有 `created_at` 和 `updated_at` 字段，由数据库自动维护（`DEFAULT CURRENT_TIMESTAMP` / `ON UPDATE CURRENT_TIMESTAMP`），**代码层面不要手动设置这两个字段**。

---

### 金额字段
**约定**：所有金额字段使用 `DECIMAL(10, 2)` 类型存储，单位为**元**。Java 层使用 `BigDecimal`，**禁止使用 `double` 或 `float`**。
